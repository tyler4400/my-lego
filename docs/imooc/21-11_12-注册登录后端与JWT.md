## 1. 概述

补齐后端账号体系：

- **21-11**：接 MongoDB（`nuxt-mongoose`）；写 User 模型；注册接口（查重 + 加密 + 落库）；登录接口（查库 + 比对密码）；前端 signup 加结果提示。
- **21-12**：签发 JWT、`setCookie` 种到浏览器；保护需要登录才能访问的接口。

craft-admin 的几个关键工程化选择（与课程不同）：

1. **密码加密放 `@my-lego/shared`**（`hashPassword`/`verifyPassword`，内部用 `bcryptjs`）—— 纯 JS、免原生编译，且 craft / backend 未来都能复用。
2. **JWT 抽到 `server/utils/jwt.ts`**（`signJwtToken`/`verifyJwtToken` + `JwtPayload` 类型 + 扩展 `H3EventContext`）。
3. **鉴权用高阶函数 `defineAuthResponseHandler`**（`server/utils/authHandler.ts`）包裹"需要登录"的接口，而不是全局中间件正则白名单 —— 哪个接口要保护一目了然。
4. **cookie 加安全选项**（`httpOnly`/`sameSite`/`secure`/`path` + 可配 `cookieName`）。

### 2.1 安装依赖与配置

```bash

pnpm --filter @my-lego/craft-admin add nuxt-mongoose jsonwebtoken

```

> ⚠️ Monorepo 差异（密码加密的归属，本节最大亮点）
> 课程在 admin 项目里直接装 `bcrypt`。craft-admin **把密码 hash/verify 放进 `@my-lego/shared`**（`packages/shared/src/crypto.ts`），用 `bcryptjs`：
>
> ```ts
> // packages/shared/src/crypto.ts
> import bcrypt from 'bcryptjs'
>
> const SALT_ROUNDS = 10
>
> export const hashPassword = async (plainPassword: string, saltRounds: number = SALT_ROUNDS): Promise<string> => {
>   return bcrypt.hash(plainPassword, saltRounds)
> }
>
> export const verifyPassword = async (plainPassword: string, passwordHash: string): Promise<boolean> => {
>   return bcrypt.compare(plainPassword, passwordHash)
> }
> ```
>
> 好处：① `bcryptjs` 纯 JS，**免 node-gyp 原生编译**，省掉 monorepo 里 `allowBuilds` 放行 `bcrypt` 的麻烦；② 加密逻辑集中在 shared，craft 编辑器 / backend 将来都能 `import { hashPassword } from '@my-lego/shared'`。`@my-lego/shared` 通过 `nuxt.config.ts` 的 alias 直连源码。

### 2.2 User 模型（nuxt-mongoose v1.1.2 的对象参数 API）

export const UserSchema = defineMongooseModel<UserDataProps>({
  name: 'User',
  schema: {
    username: { type: String, unique: true, required: true },
    password: { type: String },
    nickName: { type: String },
    picture: { type: String },
    email: { type: String },
    phoneNumber: { type: String },

    role: { type: String, default: 'normal' },

### 2.3 注册接口 `signup.post.ts`

```ts
// packages/craft-admin/server/api/users/signup.post.ts
import { userSignupSchema } from '#shared/validators/user'
import { hashPassword } from '@my-lego/shared'

export default defineEventHandler(async (event) => {
  // 1) 校验（后端其实只需 email+password，confirmPwd 传不传都行）
  const result = await runValidate(userSignupSchema, event)
  const { email, password } = result.data

  // 2) 查重（username 唯一）
  const exist = await UserSchema.findOne({ username: email }).lean()
  if (exist) {
    throw createError({ statusCode: 409, statusMessage: '该邮箱已被注册，请直接登录' })
  }

  // 3) 加密（来自 @my-lego/shared，saltRounds 从 runtimeConfig）
  const config = useRuntimeConfig(event)
  const hash = await hashPassword(password, config.bcrypt.saltRounds)

  // 4) 落库（toJSON 触发 transform 删密码）
  const newUser = await UserSchema.create({ username: email, email, password: hash })
  return newUser.toJSON()
})
```

> 💡 `runValidate`（21-6）在 `server/utils/`，自动导入，无需 import。
> 冲突错误用 **`statusMessage`**（不是 `message`）—— 它要展示给用户，而 Nitro 生产模式默认屏蔽 `message`、保留 `statusMessage`。

### 2.5 登录接口 `login.post.ts`（验密 + 签发 JWT + 种 cookie）

```ts
// packages/craft-admin/server/api/users/login.post.ts
import { userLoginSchema } from '#shared/validators/user'
import { verifyPassword } from '@my-lego/shared'

export default defineEventHandler(async (event) => {
  const result = await runValidate(userLoginSchema, event)
  const { email, password } = result.data

  const user = await UserSchema.findOne({ username: email }).exec()

  // 安全：用户不存在 / 密码错误，同一句模糊提示，防枚举
  if (!user) {
    throw createError({ statusCode: 404, statusMessage: '该用户不存在或者密码错误' })
  }
  const verifyPwd = await verifyPassword(password, user.password)
  if (!verifyPwd) {
    throw createError({ statusCode: 404, statusMessage: '该用户不存在或者密码错误' })
  }

  // 只把关键信息放进 token
  const userData = { username: user.username, _id: user._id }
  const config = useRuntimeConfig(event)
  const token = signJwtToken(userData as unknown as JwtPayload)

  // 种 cookie：过期与 JWT 一致 + 一组安全选项
  setCookie(event, config.jwt.cookieName, token, {
    maxAge: config.jwt.expiresIn,
    httpOnly: true, // JS 读不到，防 XSS 窃取
    sameSite: 'lax', // 防 CSRF
    secure: !import.meta.dev, // 生产走 https
    path: '/',
  })

  return user.toJSON()
})
```

> ⚠️ 安全差异（cookie，craft-admin 比课程更稳）
>
> - **`httpOnly: true`**：客户端 JS 读不到 token —— 这会影响前端登录态判断方式（不能再用 `useCookie('token')` 读，详见 21-13_14）。
> - `sameSite: 'lax'` 防 CSRF；`secure: !import.meta.dev` 生产强制 https；`path: '/'` 全站可用。
> - `userData as unknown as JwtPayload`：`user._id` 是 Mongoose 的 `ObjectId`，断言成 `JwtPayload`（`_id: string`）以满足类型。

### 2.6 JWT 工具抽离 `server/utils/jwt.ts`

把签发/校验集中，并就近声明 token 载荷类型 + 扩展 `H3EventContext`：

```ts
// packages/craft-admin/server/utils/jwt.ts
import type { SignOptions, VerifyOptions } from 'jsonwebtoken'
import jwt from 'jsonwebtoken'

export interface JwtPayload {
  username: string
  _id: string
}

// 扩展 h3 的 event.context.user 类型（鉴权后能拿到强类型）
declare module 'h3' {
  interface H3EventContext {
    user?: JwtPayload
  }
}

export const signJwtToken = (userData: JwtPayload, options: SignOptions = {}) => {
  const config = useRuntimeConfig()
  return jwt.sign(userData, config.jwt.secret, { expiresIn: config.jwt.expiresIn, ...options })
}

export const verifyJwtToken = (token: string, options: VerifyOptions = {}): JwtPayload => {
  const config = useRuntimeConfig()
  return jwt.verify(token, config.jwt.secret, options) as JwtPayload
}
```

> 💡 `server/utils/` 下的 `signJwtToken`/`verifyJwtToken` 自动导入；`JwtPayload` 类型也随之可用。`declare module 'h3'` 让 `event.context.user` 在所有 handler 里有类型、不用断言。

> ⚠️ 依赖版本差异（jsonwebtoken ESM）
> `jsonwebtoken` 是 CJS 包，Nitro 下 `import jwt from 'jsonwebtoken'` 一般能正常 interop；若报 default export 问题，可换现代 ESM 库 [`jose`](https://github.com/panva/jose)。

### 2.7 路由保护：高阶函数 `defineAuthResponseHandler`

> ⚠️ 架构差异（craft-admin 用高阶函数，不用全局中间件）
> 课程用 `server/middleware/auth.ts` + 正则白名单（对所有请求生效，靠正则排除 login/signup）。
> craft-admin 改用**高阶函数包装**：需要登录的接口显式用 `defineAuthResponseHandler(...)` 包起来，**哪个接口要保护一目了然**，不必维护正则、也不会误伤公开接口。

```ts
// packages/craft-admin/server/utils/authHandler.ts
import type { EventHandler } from 'h3'

// 高阶函数：在原始 handler 前后插入鉴权/统一处理
export const defineAuthResponseHandler = (handler: EventHandler) => {
  return defineEventHandler(async (event) => {
    // ===== before：鉴权 =====
    const config = useRuntimeConfig(event)
    const token = getCookie(event, config.jwt.cookieName)

    if (!token) {
      throw createError({ statusCode: 401, statusMessage: 'Unauthorized' })
    }

    try {
      const userData = verifyJwtToken(token) // 解密拿回 { username, _id }
      event.context.user = userData // 挂到 context 供后续 handler 用
    }
    catch {
      deleteCookie(event, config.jwt.cookieName) // token 伪造/过期 → 清 cookie
      throw createError({ statusCode: 401, statusMessage: 'Token 已过期' })
    }

    // ===== 执行原始 handler =====
    const response = await handler(event)

    // ===== after：可统一包裹响应 / 记录日志（暂未使用）=====
    return response
  })
}
```

**用它包裹需要登录的接口**：

```ts
// packages/craft-admin/server/api/users/current.get.ts —— 取当前登录用户
export default defineAuthResponseHandler(async (event) => {
  const currentUser = await UserSchema.findOne({ username: event.context.user?.username }).exec()
  return currentUser?.toJSON()
})
```

```ts
// packages/craft-admin/server/api/users/logout.post.ts —— 登出（清 cookie）
export default defineAuthResponseHandler((event) => {
  const config = useRuntimeConfig(event)
  deleteCookie(event, config.jwt.cookieName)
  return { success: true }
})
```

> 💡 `defineAuthResponseHandler` 在 `server/utils/` 自动导入，接口里直接用、无需 import。`login.post.ts` / `signup.post.ts` 是公开接口，仍用普通 `defineEventHandler`。

## 4. 易错点 + 关键 API 速查

### 4.1 易错点 Checklist

1. **`httpOnly: true` 后前端读不到 token**：登录态判断不能用 `useCookie('token')`，要靠 `/api/users/current` 接口（21-13_14 详述）。这是最容易忽略的连锁影响。
2. **密码不要在 admin 重复实现**：用 `@my-lego/shared` 的 `hashPassword`/`verifyPassword`，避免每个项目各写一份、还能免 bcrypt 原生编译。
3. **`defineMongooseModel` 用对象参数**：`{ name, schema, options }`，不是位置参数。
4. **鉴权用高阶函数 vs 全局中间件**：按接口保护用 `defineAuthResponseHandler` 包；"对所有请求"的需求（日志）才用 `server/middleware`。别把鉴权写成全局中间件后又忘了排除公开接口。
5. **面向用户的错误用 `statusMessage`**（生产保留），不要用 `message`（生产屏蔽）。
6. **`NUXT_MONGOOSE_URI` / `NUXT_JWT_SECRET` 命名**：嵌套 config 的环境变量是 `NUXT_` 前缀 + 路径大写下划线。
7. **JWT 只放 `username` + `_id`**，别塞整个 user。
8. **登录失败提示要模糊**（用户不存在 / 密码错误同一句），防枚举。

### 4.2 关键 API 速查

| API                                            | 作用                          | 来源                       |
| ---------------------------------------------- | ----------------------------- | -------------------------- |
| `hashPassword(pwd, saltRounds)` / `verifyPassword(plain, hash)` | 加密 / 比对密码 | `@my-lego/shared`          |
| `defineMongooseModel<T>({ name, schema, options })` | 定义模型（对象参数）    | `#nuxt/mongoose`           |
| `UserSchema.findOne(q).lean() / .exec()`       | 查询（普通对象 / Document）    | mongoose（自动导入）       |
| `signJwtToken(payload)` / `verifyJwtToken(token)` | 签发 / 校验 JWT          | `server/utils/jwt.ts`（自动导入） |
| `defineAuthResponseHandler(handler)`           | 高阶函数：鉴权后执行 handler   | `server/utils/authHandler.ts`（自动导入） |
| `setCookie / getCookie / deleteCookie(event, name, ...)` | 操作 cookie         | h3（全局）                 |
| `runValidate(schema, event)`                   | 读 body + 校验                | `server/utils/`（自动导入） |
| `event.context.user`                           | 鉴权后挂的用户（已声明类型）   | h3                         |

### 4.3 与课程相比，本节**改写**了什么

- 密码：`bcrypt` → `@my-lego/shared` 的 `hashPassword`/`verifyPassword`（bcryptjs）
- JWT：内联 → `server/utils/jwt.ts`（含 `JwtPayload` + `H3EventContext` 扩展）
- 鉴权：全局中间件正则 → 高阶函数 `defineAuthResponseHandler` 按接口包裹
- model：位置参数 → 对象参数 API
- cookie：加 `httpOnly`/`sameSite`/`secure`/`path` + 可配 `cookieName`
- 新增 `logout` 接口、`1.log.ts` 日志中间件、`test.http` 接口测试

## 1. 概述

前面把前端表单和校验做完了，本节补齐**后端账号体系**：

- **21-11**：接 MongoDB（`nuxt-mongoose`）；写 User 模型；注册接口 `signup.post.ts`（查重 + `bcrypt` 加密 + 落库）；登录接口 `login.post.ts`（查库 + `bcrypt.compare` 比对密码）；前端 signup 加结果提示框 `callbackMessage`。

## 2. 知识点 + 演示复盘

### 2.1 安装依赖

```bash
# 数据库模块
pnpm --filter @my-lego/craft-admin add nuxt-mongoose

# 加密 + JWT（含类型）
pnpm --filter @my-lego/craft-admin add bcrypt jsonwebtoken
pnpm --filter @my-lego/craft-admin add -D @types/bcrypt @types/jsonwebtoken
```

> ⚠️ Monorepo 差异（bcrypt 是原生模块，pnpm 会拦 build）
> `bcrypt` 含原生编译（node-gyp），pnpm 默认**不自动执行依赖的 build 脚本**。安装后若报 "bcrypt build was ignored"，把它加进根 `pnpm-workspace.yaml` 的 `allowBuilds`：
>
> ```yaml
> allowBuilds:
>   '@nestjs/core': true
>   bcrypt: true        # ← 新增
>   esbuild: true
>   sharp: true
>   # ...
> ```
>
> 然后 `pnpm rebuild bcrypt`（或 `pnpm approve-builds`）。
>
> 💡 **更省心的替代：`bcryptjs`**（纯 JS，免原生编译，API 与 bcrypt 基本一致）。craft-admin 想避开编译坑可以装 `bcryptjs`，代码只需把 `import bcrypt from 'bcrypt'` 换成 `import bcrypt from 'bcryptjs'`。下面主线用 `bcrypt`（与课程一致）。

> ⚠️ 依赖版本差异（jsonwebtoken 的 ESM 坑）
> `jsonwebtoken` 是 CJS 包，在 Nitro 里偶尔报 `does not provide an export named 'default'`。若遇到，两条路：① 在 `nuxt.config.ts` 加 `nitro: { moduleSideEffects: [], }` 并用 `import jwt from 'jsonwebtoken'`（多数情况 Nitro 的 CJS interop 能处理）；② 换现代 ESM 库 [`jose`](https://github.com/panva/jose)。主线用 `jsonwebtoken`（与课程一致）。

### 2.2 配置 MongoDB（nuxt-mongoose）

> ⚠️ 依赖版本差异（参考第 20 章 hello-nuxt 的 `nuxt-mongoose` 笔记）
> `nuxt-mongoose` 当前 **v1.1.2 兼容 Nuxt 4**，但与课程有两处实操差异：① 连接串环境变量用 **`NUXT_MONGOOSE_URI`**；② 模型里 `defineMongooseModel` 推荐**显式** `import ... from '#nuxt/mongoose'`。

`nuxt.config.ts` 注册模块（与已有的 vee-validate 模块并列）：

```ts
// packages/craft-admin/nuxt.config.ts（节选）
export default defineNuxtConfig({
  modules: [
    'nuxt-mongoose',
    ['@vee-validate/nuxt', { /* ...21-7 的配置... */ }],
  ],

  // 运行时配置：服务端私有 + 公开
  runtimeConfig: {
    jwt: {
      secret: '', // 被 .env 的 NUXT_JWT_SECRET 覆盖
      expiresIn: 60 * 60, // 1 小时（秒）
    },
    bcrypt: {
      saltRounds: 10,
    },
  },
})
```

`.env`（项目根 `packages/craft-admin/.env`，加进 `.gitignore`）：

```bash
# nuxt-mongoose 自动读取该变量作为连接串
NUXT_MONGOOSE_URI=mongodb://localhost:27017/craft-admin

# 覆盖 runtimeConfig.jwt.secret（嵌套 key → NUXT_JWT_SECRET）
NUXT_JWT_SECRET=replace-with-your-own-secret
```

> ⚠️ Nuxt 4 差异（嵌套 runtimeConfig 的环境变量命名）
> `runtimeConfig.jwt.secret` 对应的环境变量是 **`NUXT_JWT_SECRET`**（前缀 `NUXT_` + 路径大写下划线连接）。写错成 `JWT_SECRET` 不会生效。

连接成功后终端会打印 `connected to mongodb`。

### 2.3 User 模型

`server/models/user.ts`（`nuxt-mongoose` 默认扫描 `server/models/`）：

```ts
// packages/craft-admin/server/models/user.ts
import { defineMongooseModel } from '#nuxt/mongoose'

export interface UserDataProps {
  username: string
  password: string
  email?: string
  nickName?: string
  picture?: string
  phoneNumber?: string
  createdAt: string // 注意：string，不是 Date（21-13 前端要用）
  updatedAt: string
  type: 'email' | 'cellphone' | 'oauth'
  role?: 'admin' | 'normal'
}

export const UserSchema = defineMongooseModel<UserDataProps>('User', {
  username: { type: String, unique: true, required: true },
  password: { type: String },
  nickName: { type: String },
  picture: { type: String },
  email: { type: String },
  phoneNumber: { type: String },
  type: { type: String, default: 'email' },
  role: { type: String, default: 'normal' },
}, {
  timestamps: true, // 自动维护 createdAt / updatedAt
  toJSON: {
    // 返回给前端时删掉敏感字段
    transform(_doc, ret) {
      delete ret.password
      delete ret.__v
    },
  },
})
```

> ⚠️ Nuxt 4 / Monorepo 差异
>
> - **model 不能放 `shared/`**：它 import `#nuxt/mongoose`（Nitro 专属），而 `shared/` 不能含 Nitro 代码。model 属于服务端，放 `server/models/`。
> - `defineMongooseModel` 用 **`#nuxt/mongoose`** 显式 import（课程靠自动导入直接用）。
> - `toJSON.transform` 删 `password` —— 后面所有 `.toJSON()` 返回都自动隐藏密码。

### 2.4 注册接口 `signup.post.ts`（21-11）

```ts
// packages/craft-admin/server/api/users/signup.post.ts
import { runValidate } from '#server/utils/runValidate'
import { userSignupSchema } from '#shared/validators/user'
import bcrypt from 'bcrypt'
import { UserSchema } from '~~/server/models/user'

export default defineEventHandler(async (event) => {
  const config = useRuntimeConfig()

  // 1) 校验（复用 21-6 的 runValidate；后端其实只需要 email+password，
  //    这里仍用 signup schema，confirmPwd 传不传都行）
  const result = await runValidate(userSignupSchema, event)
  const { email, password } = result.data

  // 2) 查重：username 唯一
  const exist = await UserSchema.findOne({ username: email }).lean()
  if (exist) {
    throw createError({
      statusCode: 409, // Conflict
      statusMessage: '该邮箱已被注册，请直接登录',
    })
  }

  // 3) 加密密码
  const hash = await bcrypt.hash(password, config.bcrypt.saltRounds)

  // 4) 落库
  const newUser = await UserSchema.create({ username: email, password: hash, email })
  return newUser
})
```

> ⚠️ 关键：注册冲突用 `statusMessage` 而非 `message`
> 这条错误要**展示给用户**。Nitro 生产模式默认屏蔽 `message`，但保留 `statusMessage`，所以面向用户的提示一律用 `statusMessage`（见 21-5_6 §4.1 第 6 条）。`runValidate` 里的 `message: '验证失败'` 是给开发看的、字段错误走 `data`，无所谓。

> ⚠️ Nuxt 4 差异（import model 的路径）
> `~~` 指向项目根（craft-admin 根），所以 `~~/server/models/user`。也可用相对路径。`#shared`、`#server` 别名同 21-5/21-6。

### 2.6 登录接口比对密码（21-11 后半）

把 21-6 的 `login.post.ts` 占位逻辑替换为真实查库 + 比对：

```ts
// packages/craft-admin/server/api/users/login.post.ts（21-11 版，JWT 在 2.7 补）
import { runValidate } from '#server/utils/runValidate'
import { userLoginSchema } from '#shared/validators/user'
import bcrypt from 'bcrypt'
import { UserSchema } from '~~/server/models/user'

export default defineEventHandler(async (event) => {
  const result = await runValidate(userLoginSchema, event)
  const { email, password } = result.data

  const user = await UserSchema.findOne({ username: email }).exec()

  // 安全：用户不存在 / 密码错误，给同一句模糊提示，避免泄漏"账号是否存在"
  if (!user) {
    throw createError({ statusCode: 404, statusMessage: '该用户不存在或者密码错误' })
  }
  const verifyPwd = await bcrypt.compare(password, user.password)
  if (!verifyPwd) {
    throw createError({ statusCode: 404, statusMessage: '该用户不存在或者密码错误' })
  }

  // 21-12 在这里签发 JWT + 种 cookie
  return user.toJSON()
})
```

> 💡 安全细节：用户不存在和密码错误**返回完全相同的提示**，防止攻击者通过差异枚举出哪些邮箱已注册。
> 注意这里 `findOne(...).exec()` 拿到的是完整 Document（能调 `.toJSON()`，触发 model 里的 transform 删密码）；查重时用 `.lean()` 拿普通对象更轻量。

### 2.7 签发 JWT + 种 Cookie（21-12）

在 `login.post.ts` 验证通过后补上 JWT：

```ts
// packages/craft-admin/server/api/users/login.post.ts（21-12 最终版）
import { runValidate } from '#server/utils/runValidate'
import { userLoginSchema } from '#shared/validators/user'
import bcrypt from 'bcrypt'
import jwt from 'jsonwebtoken'
import { UserSchema } from '~~/server/models/user'

export default defineEventHandler(async (event) => {
  const config = useRuntimeConfig()
  const result = await runValidate(userLoginSchema, event)
  const { email, password } = result.data

  const user = await UserSchema.findOne({ username: email }).exec()
  if (!user) {
    throw createError({ statusCode: 404, statusMessage: '该用户不存在或者密码错误' })
  }
  const verifyPwd = await bcrypt.compare(password, user.password)
  if (!verifyPwd) {
    throw createError({ statusCode: 404, statusMessage: '该用户不存在或者密码错误' })
  }

  // 只把关键信息放进 token（不要塞整个 user）
  const userData = { username: user.username, _id: user._id }
  const token = jwt.sign(userData, config.jwt.secret, { expiresIn: config.jwt.expiresIn })

  // 种 cookie，过期时间与 JWT 一致
  setCookie(event, 'token', token, { maxAge: config.jwt.expiresIn })

  // toJSON 触发 transform 删密码
  return user.toJSON()
})
```

要点：

- **JWT 里只放 `username` + `_id`** —— 最小必要信息，解密后能定位用户即可。
- `setCookie(event, key, value, options)` 是 h3 全局函数；`maxAge` 单位秒，与 `jwt expiresIn` 对齐，两者同时过期。
- cookie 种好后，同域请求浏览器会自动带上，无需前端手动处理。

### 2.8 保护路由：服务端中间件 `server/middleware/auth.ts`（21-12）

需求：`/api/users/*` 里**只有 `login` 和 `signup` 不需要登录**，其余都要校验 token。

```ts
// packages/craft-admin/server/middleware/auth.ts
import jwt from 'jsonwebtoken'

export default defineEventHandler((event) => {
  // 只拦 /api/users/ 下、且不是 login / signup 的请求
  const regex = /^\/api\/users\/(?!login$|signup$).*/
  if (!regex.test(event.path)) {
    return
  }

  const config = useRuntimeConfig()
  const token = getCookie(event, 'token')

  if (!token) {
    throw createError({ statusCode: 401, statusMessage: 'Unauthorized' })
  }

  try {
    // 解密，拿回签发时存的 { username, _id }
    const userData = jwt.verify(token, config.jwt.secret)
    // 挂到 event.context，供后续 handler 使用
    event.context.user = userData
  }
  catch {
    // token 伪造或过期：清 cookie + 报错
    deleteCookie(event, 'token')
    throw createError({ statusCode: 401, statusMessage: 'Token 已过期' })
  }
})
```

要点：

- **`server/middleware/` 的中间件对每个请求都跑**，所以要用正则过滤出"需要保护"的路由。
- `getCookie` / `deleteCookie` 与 `setCookie` 成对，都是 h3 全局函数。
- `jwt.verify` 失败（伪造/过期）会抛异常 → catch 里清 cookie + 报错。
- 校验通过把解密结果挂 **`event.context.user`** —— 21-14 的 `current` 接口就从这里取 `username` 查完整用户信息。

> 💡 状态码用 401（Unauthorized）比课程的 400 更语义化（未认证）。课程用了 400，按你喜好选；本文用 401。

用 Postman 测：登录拿到 cookie → 请求 `/api/users/current`（受保护）自动带 cookie → 通过；删掉/改坏 cookie → 401。

## 4. 易错点 + 关键 API 速查

### 4.1 易错点 Checklist

1. **`NUXT_MONGOOSE_URI` / `NUXT_JWT_SECRET` 命名**：嵌套 runtimeConfig 的环境变量是 `NUXT_` 前缀 + 路径大写下划线。写错不生效且不报错。
2. **bcrypt build 被 pnpm 拦**：加 `allowBuilds: { bcrypt: true }` + `pnpm rebuild bcrypt`；嫌麻烦直接用 `bcryptjs`。
3. **面向用户的错误用 `statusMessage`**，不要用 `message`（生产被屏蔽）。
4. **JWT 别塞整个 user**：只放 `username` + `_id`，减小 token、避免泄漏。
5. **model 不能放 `shared/`**：它依赖 `#nuxt/mongoose`（Nitro），放 `server/models/`。
6. **`server/middleware` 对所有请求生效**：必须正则过滤，否则把 login/signup 也拦了就没法登录。
7. **登录失败提示要模糊**：用户不存在 / 密码错误返回同一句，防枚举。
8. **`.lean()` vs `.exec()`**：查重用 `.lean()`（普通对象，轻量）；要 `.toJSON()` 触发删密码 transform 时用 `.exec()` 拿 Document。
9. **`jsonwebtoken` 的 ESM 报错**：Nitro 下若 `import jwt from 'jsonwebtoken'` 报 default export 问题，换 `jose` 或调 nitro 配置。
10. **`.env` 别提交**：加 `.gitignore`；`NUXT_JWT_SECRET` 是敏感信息。

### 4.2 关键 API 速查

| API                                            | 作用                          | 来源                       |
| ---------------------------------------------- | ----------------------------- | -------------------------- |
| `defineMongooseModel<T>(name, schema, opts)`   | 定义 Mongoose 模型             | `#nuxt/mongoose`           |
| `UserSchema.findOne(query).lean() / .exec()`   | 查询（普通对象 / Document）    | mongoose                   |
| `UserSchema.create(data)`                      | 新建文档                       | mongoose                   |
| `doc.toJSON()`                                 | 序列化（触发 transform 删密码） | mongoose                   |
| `bcrypt.hash(pwd, saltRounds)`                 | 加密密码                       | bcrypt                     |
| `bcrypt.compare(plain, hash)`                  | 比对密码                       | bcrypt                     |
| `jwt.sign(payload, secret, { expiresIn })`     | 签发 token                     | jsonwebtoken               |
| `jwt.verify(token, secret)`                    | 校验/解密 token                | jsonwebtoken               |
| `setCookie / getCookie / deleteCookie(event, ...)` | 操作 cookie                | h3（全局）                 |
| `useRuntimeConfig()`                           | 取 runtimeConfig               | Nuxt（全局）               |
| `event.context.user = ...`                     | 跨中间件→handler 传数据         | h3                         |
| `event.path`                                   | 当前请求路径（中间件过滤用）    | h3                         |

### 4.3 与课程相比，本节**改写**了什么

- 连接串 `MONGODB_URI` → `NUXT_MONGOOSE_URI`；model 显式 `#nuxt/mongoose`

- bcrypt build 需 monorepo `allowBuilds`（或换 bcryptjs）
- 受保护路由状态码 400 → 401（更语义化）
- 面向用户错误统一 `statusMessage`

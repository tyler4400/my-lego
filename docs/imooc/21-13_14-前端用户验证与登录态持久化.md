## 2. 知识点 + 演示复盘

### 2.1 任务一：改造 `useCurrentUser` 数据结构（21-13）

之前测试用的 `currentUser` 只存了个 `username`，现在要存后端返回的完整用户对象。

#### 把用户类型提到 `shared/types/`（Nuxt 4 关键适配）

> ⚠️ Nuxt 4 差异（为什么类型要放 `shared/types/`）
> 课程（Nuxt 3）把 `UserProps` 放 `types/user.ts`，里面 `import { UserDataProps } from '~/server/models/user'`。
> Nuxt 4 **把 app 和 server 拆成了独立的 TypeScript 工程**（`tsconfig.app.json` / `tsconfig.server.json`），app 里直接 import `server/` 的类型可能报"找不到模块"。
> 解决：把两端共用的类型放进 **`shared/types/`**（Nuxt 4 会自动导入，且 app/server 两个 tsconfig 都包含它）。

新建 `packages/craft-admin/shared/types/user.ts`：

```ts
// packages/craft-admin/shared/types/user.ts
// shared/types 下的类型会被自动导入（app + server 都能直接用）

export interface UserDataProps {
  username: string
  password: string
  email?: string
  nickName?: string
  picture?: string
  phoneNumber?: string
  createdAt: string
  updatedAt: string
  type: 'email' | 'cellphone' | 'oauth'
  role?: 'admin' | 'normal'
}

// 前端登录态结构
export interface UserProps {
  isLogin: boolean
  data: UserDataProps | null
}
```

把 21-11_12 里写在 `server/models/user.ts` 的 `UserDataProps` **移到这里**，model 改为引用它：

```ts
// packages/craft-admin/server/models/user.ts（改：UserDataProps 来自 shared/types）
import { defineMongooseModel } from '#nuxt/mongoose'
import type { UserDataProps } from '#shared/types/user'

export const UserSchema = defineMongooseModel<UserDataProps>('User', {
  /* ...字段定义不变... */
})
```

> 💡 `shared/types/` 自动导入意味着 `UserDataProps` / `UserProps` 在 app 与 server 里**不写 import 也能用**；但显式 `import type { ... } from '#shared/types/user'` 更利于 IDE 与可读性，按喜好选。

#### 改 `useCurrentUser`

> ⚠️ Nuxt 4 差异（composables 位置）
> 课程在项目根 `composables/`；craft-admin 在 **`app/composables/`**。

```ts
// packages/craft-admin/app/composables/useCurrentUser.ts
import type { UserProps } from '#shared/types/user'

export const useCurrentUser = () => {
  // useState 是 Nuxt 的跨组件、SSR 友好的全局状态
  return useState<UserProps>('currentUser', () => ({
    isLogin: false,
    data: null,
  }))
}
```

#### 登录成功后写入 `currentUser`

回到 `login.vue`，登录成功把返回的用户写进 `currentUser`、并跳首页：

```ts
// login.vue <script setup> 里的 handleLogin（21-13 增补）
const currentUser = useCurrentUser()

const handleLogin = handleSubmit(async () => {
  try {
    const data = await $fetch('/api/users/login', { method: 'POST', body: values })
    currentUser.value.isLogin = true
    currentUser.value.data = data as UserDataProps
    navigateTo('/')
  }
  catch (error) {
    console.error('登录失败', error)
  }
})
```

> 💡 临时 Header 显示用户名：21-3 的 `default.vue` 里那个占位 Header，现在可以显示 `currentUser.data?.username`（21-17 做正式 `GlobalHeader` 时再抽组件）。

### 2.2 任务二：全局中间件保护所有页面（21-13）

需求：除 `/login`、`/signup` 外，所有路由都要登录。用**全局路由中间件**（`.global` 后缀，自动对每个路由生效，不用每页 `definePageMeta`）。

> ⚠️ Nuxt 4 差异（middleware 位置）
> 课程在项目根 `middleware/`；craft-admin 在 **`app/middleware/`**。全局中间件文件名要带 `.global`。

```ts
// packages/craft-admin/app/middleware/auth.global.ts（21-13 基础版，21-14 会补持久化）
export default defineNuxtRouteMiddleware((to) => {
  // 登录 / 注册页放行
  if (to.path === '/login' || to.path === '/signup') {
    return
  }

  const token = useCookie('token')
  if (!token.value) {
    return navigateTo('/login')
  }
})
```

此时：未登录访问任意页面（如 `/`、`/users/1`）都会跳 `/login`；登录后能进。但**刷新页面登录态会丢**（`currentUser` 是内存状态）—— 这就是任务三要解决的。

### 2.3 任务三 Step 1：新增 `/api/users/current` 接口（21-14）

JWT 里只存了 `username` + `_id`，信息太少。前端持久化时需要完整用户信息，所以加一个接口：从 token 解出的 `username` 查库返回完整用户。

```ts
// packages/craft-admin/server/api/users/current.get.ts
import { UserSchema } from '~~/server/models/user'

export default defineEventHandler(async (event) => {
  // event.context.user 是 21-12 的 auth 中间件 jwt.verify 后挂上的
  const { username } = event.context.user as { username: string }
  const currentUserData = await UserSchema.findOne({ username }).exec()
  return currentUserData?.toJSON()
})
```

> 💡 `current.get.ts` 在 `/api/users/` 下且不是 login/signup，所以会被 21-12 的 `auth.ts` 中间件保护：先校验 token、挂 `event.context.user`，再轮到这个 handler。
>
> 想让 `event.context.user` 有类型（去掉 `as` 断言），可在 `shared/types` 或 server 端用模块扩展声明 `declare module 'h3' { interface H3EventContext { user: { username: string, _id: string } } }`。本节先用断言。

### 2.4 任务三 Step 2：中间件里恢复登录态（21-14 核心）

给 `auth.global.ts` 补持久化逻辑：

```ts
// packages/craft-admin/app/middleware/auth.global.ts（21-14 完整版）
import type { UserDataProps } from '#shared/types/user'

export default defineNuxtRouteMiddleware(async (to) => {
  if (to.path === '/login' || to.path === '/signup') {
    return
  }

  const currentUser = useCurrentUser()
  const token = useCookie('token')

  // 1) 没有 token → 未登录，去登录页
  if (!token.value) {
    return navigateTo('/login')
  }

  // 2) 有 token 但内存里还没登录态 → 第一次进入应用（含刷新），向后端验证并恢复
  if (token.value && !currentUser.value.isLogin) {
    // SSR 首次渲染时，useFetch 不会自动带浏览器 cookie，需要手动透传
    const cookie = useRequestHeader('cookie') as string

    const { data, error } = await useFetch<UserDataProps>('/api/users/current', {
      headers: {
        cookie, // 把请求头里的 cookie 透传给内部接口
        accept: 'application/json', // 否则可能返回整页 HTML
      },
    })

    if (!error.value) {
      currentUser.value.isLogin = true
      currentUser.value.data = data.value
    }
    else {
      // token 失效/过期：清状态 + 清 cookie + 去登录
      currentUser.value.isLogin = false
      token.value = null
      return navigateTo('/login')
    }
  }
})
```

### 2.5 任务三的两个 SSR 关键点（讲师重点强调）

#### 为什么用 `useFetch` 而不是 `$fetch`

中间件在 **SSR 首次渲染**时运行在服务器端。这里用 `useFetch` 调本应用的 `/api/users/current`，Nuxt 足够聪明：**直接调用后端路由逻辑，不发额外 HTTP 请求**，更快。

#### 为什么要手动设 `headers`

SSR 首次渲染时，`useFetch` 发起的内部调用**不会自动带上浏览器的 cookie**（此时浏览器渲染还没开始）。而我们的 `/api/users/current` 受 `auth` 中间件保护、要读 cookie 里的 token。所以必须：

- `cookie: useRequestHeader('cookie')` —— 把进来的请求头里的 cookie 透传过去（否则后端读不到 token，401）。
- `accept: 'application/json'` —— 不加的话默认 `accept` 可能是 HTML，后端会返回整页 HTML 而不是 JSON。

> ⚠️ Nuxt 4 差异
> `useCookie` / `useRequestHeader` / `useFetch` / `useState` 都是 Nuxt 自动导入的全局 composable，用法与 Nuxt 3 一致；位置上 composables / middleware 都在 `app/` 下。

### 2.6 验证整套流程

1. 登录成功 → 跳首页，看到用户名（临时 Header）。
2. **刷新页面** → 登录态还在（中间件用 cookie 里的 token 调 current 自动恢复），不再跳回登录。
3. 打开 DevTools 改坏 cookie 的 token → 刷新 → current 返回 401 → 中间件清 token、跳 `/login`。
4. 未登录直接访问 `/` 或任意页 → 跳 `/login`。

> 💡 体会"全栈框架"的妙处：SSR 首次渲染时登录态在**服务端**就恢复好了，渲染到浏览器时已是登录状态，没有"先白屏再请求"的 SPA 式闪烁。

## 4. 易错点 + 关键 API 速查

### 4.1 易错点 Checklist

1. **SSR 下 `useFetch` 必须手动透传 `cookie`**：否则内部调用不带 token，受保护接口返回 401，导致刚登录就被踢。
2. **`accept: 'application/json'` 不能漏**：漏了后端可能返回整页 HTML，前端解析出错。
3. **类型放 `shared/types/`**：Nuxt 4 app 不能直接 import `server/` 的类型（tsconfig 分离），共用类型放 `shared/types/`（自动导入）。
4. **`useCookie('token')` 返回 ref**：取值要 `.value`；置空登出写 `token.value = null`（不是 `useCookie(...)= null`）。
5. **中间件里 `navigateTo` 要 `return`**：`return navigateTo('/login')`，不 return 可能继续往下执行。
6. **全局中间件文件名要带 `.global`**：`auth.global.ts`，否则不会自动对所有路由生效。
7. **判断"第一次进入"用 `token.value && !currentUser.value.isLogin`**：避免每次路由跳转都重新请求 current（登录态已在内存就跳过）。
8. **`event.context.user` 无类型**：用断言或声明 `H3EventContext` 模块扩展。
9. **`createdAt`/`updatedAt` 用 `string`**：JSON 序列化后是字符串，类型定义对齐，避免 `Date` 类型不匹配。

### 4.2 关键 API 速查

| API                                   | 作用                                   | 端           |
| ------------------------------------- | -------------------------------------- | ------------ |
| `useState<T>(key, init)`              | SSR 友好的全局状态（`currentUser`）     | 客户端/同构  |
| `useCookie(key)`                      | 读写 cookie（返回 ref，`.value`）       | 客户端/同构  |
| `useRequestHeader('cookie')`          | 读取当前请求头（SSR 透传 cookie 用）    | 服务端渲染   |
| `useFetch<T>(url, { headers })`       | SSR 友好请求；内部调用不发额外 HTTP     | 同构         |
| `defineNuxtRouteMiddleware(fn)`       | 定义路由中间件                          | 客户端/同构  |
| `navigateTo(path)`                    | 编程式跳转（中间件里要 `return`）        | 同构         |
| `event.context.user`                  | 取 auth 中间件挂的解密用户              | 服务端       |
| `doc.toJSON()`                        | 序列化（删密码）                        | 服务端       |

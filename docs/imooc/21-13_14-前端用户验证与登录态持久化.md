## 1. 概述

后端鉴权（21-12）做完，本节在前端完成登录态管理，三个任务：

1. **改 `useCurrentUser` 数据结构** —— 存后端返回的完整用户 `{ isLogin, data }`。
2. **全局路由中间件保护页面** —— 除 `/login`、`/signup` 外都要登录。
3. **持久化登录态** —— 刷新后不丢，靠 `/api/users/current` 接口在中间件里恢复。

> ⚠️ httpOnly 带来的范式改变（贯穿全篇）
> 课程 cookie 没设 `httpOnly`，前端能 `useCookie('token')` 读 token 来判断"有没有登录"。
> craft-admin 的 token cookie 是 **`httpOnly`** 的（21-12，防 XSS），**前端 JS 读不到**。所以判断登录态只能靠：① 内存里的 `currentUser.isLogin`；② 调用受保护的 `/api/users/current`（带不带得上有效 cookie 由浏览器/SSR 透传决定）。这是 craft-admin 与课程在前端逻辑上的根本区别。

## 2. 知识点 + 演示复盘

### 2.1 任务一：`useCurrentUser` 数据结构

用户类型放在 `shared/types/`（21-11 已建好 `UserDataProps`，这里补前端态 `UserProps`）：

```ts
// packages/craft-admin/shared/types/user.ts（自动导入，app + server 通用）
export interface UserDataProps {
  username: string
  password: string
  email?: string
  // ...其余字段见 21-11
  createdAt: string
  updatedAt: string
  type: 'email' | 'cellphone' | 'oauth'
  role?: 'admin' | 'normal'
}

// 前端登录态结构
export interface UserProps {
  isLogin: boolean
  data: UserDataProps | null
}
```

> ⚠️ Nuxt 4 差异（类型为什么放 `shared/types/`）
> Nuxt 4 把 app 与 server 拆成独立 TS 工程（`tsconfig.app.json` / `tsconfig.server.json`），app 里直接 import `server/` 的类型会"找不到模块"。放 **`shared/types/`**（自动导入，两个 tsconfig 都含）即可两端通用。`server/models/user.ts` 的 `defineMongooseModel<UserDataProps>` 也是引用这里的类型。

```ts
// packages/craft-admin/app/composables/useCurrentUser.ts
export const useCurrentUser = () => {
  // useState：SSR 友好的跨组件全局状态
  return useState<UserProps>('currentUser', () => ({
    isLogin: false,
    data: null,
  }))
}
```

> ⚠️ Nuxt 4 差异（composables 位置）
> 课程在项目根 `composables/`；craft-admin 在 **`app/composables/`**。`UserProps` 自动导入（来自 `shared/types/`），这里不写 import 也能用。

### 2.2 任务一（续）：登录成功写入 `currentUser`

登录成功后，直接用接口返回的用户填充 `currentUser` 并跳首页：

```ts
// login.vue <script setup>（21-13 增补；template 在前略）
const currentUser = useCurrentUser()

const handleLogin = handleSubmit(async () => {
  try {
    const data = await $fetch('/api/users/login', { method: 'POST', body: values })
    // 直接用返回数据填充内存态：首页中间件看到 isLogin=true 就直接放行，省一次 current 请求
    currentUser.value.isLogin = true
    currentUser.value.data = data as UserDataProps
    navigateTo('/')
  }
  catch (error) {
    console.error('登录失败', error)
  }
})
```

> 💡 登录成功**直接写内存态**是和 2.3 中间件"内存优先"逻辑的最佳配合：跳首页时 `isLogin` 已是 `true`，中间件直接放行，不必再请求一次 `/current`。

### 2.3 任务二 + 三：全局中间件（保护 + 持久化）

craft-admin 的 `auth.global.ts` 把"保护"和"持久化"合到一处，**不读 cookie**（httpOnly 读不到），改用"内存态优先，否则请求 `/current`"：

```ts
// packages/craft-admin/app/middleware/auth.global.ts
export default defineNuxtRouteMiddleware(async (to) => {
  // 登录 / 注册页放行
  if (to.path === '/login' || to.path === '/signup') {
    return
  }

  const currentUser = useCurrentUser()

  // 1) 内存里已是登录态 → 直接放行（客户端路由跳转走这里，不调接口）
  if (currentUser.value.isLogin) {
    return
  }

  // 2) 没有内存态（首次进入 / 刷新 / 客户端首跳）→ 请求 /current 验证并恢复
  // SSR 首次渲染时 useFetch 不会自动带浏览器 cookie，需手动透传（httpOnly 服务端读得到）
  const headers = useRequestHeaders(['cookie']) // 服务端：{ cookie: 'xxx' }；客户端：{}
  const { data, error } = await useFetch<UserDataProps>('/api/users/current', {
    headers: {
      ...headers, // 客户端为空对象，不会覆盖浏览器自动携带的 cookie
      accept: 'application/json',
    },
  })

  if (!error.value && data.value) {
    currentUser.value.isLogin = true
    currentUser.value.data = data.value
    return
  }

  // 无有效 token / 已过期 → /current 返回 401 → 去登录页
  return navigateTo('/login')
})
```

> ⚠️ 与课程的核心差异（务必理解）
>
> | | 课程（cookie 非 httpOnly） | craft-admin（httpOnly） |
> | --- | --- | --- |
> | "有没有 token" | `useCookie('token').value` 前端直接读 | **读不到**，不用这招 |
> | 未登录判断 | 没 token → 跳登录 | 内存没登录态 → 请求 `/current`，401 才跳登录 |
> | 恢复登录态 | token 存在但内存没登录 → 请求 current | 内存没登录 → 请求 current（统一入口） |
>
> craft-admin 的逻辑更简洁：**不关心 cookie 本身，只问后端"我现在算不算登录"**（`/current` 受 `defineAuthResponseHandler` 保护，没 token/失效就 401）。

### 2.4 `/api/users/current` 接口（21-11_12 已建，这里回顾）

它就是登录态恢复的数据源 —— 从鉴权中间件解出的 `username` 查完整用户：

```ts
// packages/craft-admin/server/api/users/current.get.ts
export default defineAuthResponseHandler(async (event) => {
  const currentUser = await UserSchema.findOne({ username: event.context.user?.username }).exec()
  return currentUser?.toJSON()
})
```

> 💡 用 `defineAuthResponseHandler`（21-12）包裹：内部先校验 token、把 `{ username, _id }` 挂到 `event.context.user`，再执行此 handler。没有有效 token 直接 401，前端 `error.value` 据此跳登录。

### 2.5 两个 SSR 关键点

#### 为什么用 `useFetch` 而不是 `$fetch`

中间件在 **SSR 首次渲染**时跑在服务端。用 `useFetch` 调本应用的 `/current`，Nuxt 会**直接调用后端逻辑、不发额外 HTTP 请求**，更快，且首屏渲染前登录态就恢复好了（没有 SPA 式"先白屏再请求"的闪烁）。

#### 为什么手动透传 `cookie`

SSR 首次渲染时，`useFetch` 的内部调用**不会自动带浏览器 cookie**。而 `/current` 受保护、要读 httpOnly cookie 里的 token，所以：

- `...useRequestHeaders(['cookie'])` —— 把进来的请求头 cookie 透传过去（服务端读得到 httpOnly cookie）。
- `accept: 'application/json'` —— 不加可能返回整页 HTML。
- 客户端导航时 `useRequestHeaders(['cookie'])` 返回 `{}`，不影响浏览器对同域请求自动携带 cookie。

> ⚠️ Nuxt 4 差异
> `useState` / `useRequestHeaders` / `useFetch` / `navigateTo` 都是自动导入的全局 composable；composables / middleware 都在 `app/` 下。课程用单数 `useRequestHeader('cookie')`，craft-admin 用复数 **`useRequestHeaders(['cookie'])`**（返回对象，便于 `...` 展开合并）。

### 2.6 验证

1. 登录成功 → 跳首页（内存态已写，中间件直接放行）。
2. **刷新** → 中间件内存态丢失 → SSR 透传 cookie 请求 `/current` → 恢复登录态，不跳登录。
3. DevTools 删/改坏 cookie → 刷新 → `/current` 401 → 跳 `/login`。
4. 未登录直接访问任意页 → 中间件请求 `/current` 401 → 跳 `/login`。

## 4. 易错点 + 关键 API 速查

### 4.1 易错点 Checklist

1. **httpOnly cookie 前端读不到**：别再用 `useCookie('token')` 判断登录态（会永远是 `null`）。用内存态 + `/current`。
2. **SSR 下 `useFetch` 必须透传 `cookie`**：否则内部 `/current` 调用不带 token，刚登录就被踢。
3. **`accept: 'application/json'` 不能漏**：否则可能返回整页 HTML。
4. **类型放 `shared/types/`**：app 不能直接 import `server/` 的类型。
5. **中间件里 `navigateTo` 要 `return`**。
6. **全局中间件文件名带 `.global`**。
7. **登录成功直接写内存态**：避免跳首页后中间件再请求一次 `/current`（虽不影响功能，但多一次请求）。
8. **`useRequestHeaders` 是复数**，参数是数组 `['cookie']`，返回对象。

### 4.2 关键 API 速查

| API                                   | 作用                                   | 端           |
| ------------------------------------- | -------------------------------------- | ------------ |
| `useState<T>(key, init)`              | SSR 友好全局状态（`currentUser`）       | 客户端/同构  |
| `useRequestHeaders(['cookie'])`       | 读当前请求头（SSR 透传 cookie）         | 服务端渲染   |
| `useFetch<T>(url, { headers })`       | SSR 友好请求；内部调用不发额外 HTTP     | 同构         |
| `defineNuxtRouteMiddleware(fn)`       | 定义路由中间件                          | 客户端/同构  |
| `navigateTo(path)`                    | 编程式跳转（中间件里 `return`）          | 同构         |
| `defineAuthResponseHandler(handler)`  | 受保护接口的高阶函数（21-12）            | 服务端       |
| `event.context.user`                  | 鉴权后挂的用户（jwt.ts 已声明类型）      | 服务端       |

### 4.3 与课程相比，本节**改写**了什么

- 登录态判断：`useCookie('token')` → **内存 `isLogin` 优先 + 请求 `/current`**（因 httpOnly）
- `useRequestHeader('cookie')` → `useRequestHeaders(['cookie'])`（复数）
- 用户类型：`types/` import server → `shared/types/`（自动导入）
- `/current` 用 `defineAuthResponseHandler` 保护（21-12 的高阶函数）
- 登录成功直接写内存态 + 跳首页

## 1. 概述

21-4 学完了 Zod 的 API，本节把 Zod 真正接入登录流程：

- **21-5（客户端）**：在 `login.vue` 用 `reactive` form data 配合 `safeParse` 做提交前校验；用 `treeifyError` 把错误格式化成对象，按字段显示在每个 input 下方。

**关键架构决策（craft-admin 独有）**：把 `validators/user.ts` 放到 **`@my-lego/shared`** 包里，让客户端和服务端真正使用同一份 schema —— 这是 monorepo 的天然红利，比课程的"客户端服务端各自 import 同一个文件"更彻底。

### 2.2 Step 2 —— 客户端：21-5 在 `login.vue` 接入校验

> 💡 21-3 已经创建过 `app/pages/login.vue` 的占位。本节把它改造成真正的登录表单。**这次我们不引入 HyperUI 模板**（21-2 提到过，HyperUI v3 class 与 v4 有差异要替换）—— 写一个最简洁的表单，21-9 完成 VeeValidate + ValidateInput 抽组件后再考虑要不要美化。

```vue
<!-- packages/craft-admin/app/pages/login.vue -->
<script setup lang="ts">
import { userLoginSchema, type UserLoginType } from '@my-lego/shared'

definePageMeta({ layout: 'auth' })

// reactive form data
const formData = reactive<UserLoginType>({
  email: '',
  password: '',
})

// 错误信息（按字段挂）
const emailError = ref<string[] | undefined>(undefined)
const passwordError = ref<string[] | undefined>(undefined)

const handleLogin = () => {
  const result = userLoginSchema.safeParse(formData)

  if (!result.success) {
    // ⚠️ v4 用 treeifyError 替代 v3 的 error.format()
    const tree = z.treeifyError(result.error)
    emailError.value = tree.properties?.email?.errors
    passwordError.value = tree.properties?.password?.errors
    return
  }

  // 校验通过 → 重置错误 → 发请求（21-6 加 server 端时联调）
  emailError.value = undefined
  passwordError.value = undefined
  console.log('valid', result.data)
}
</script>

<template>
  <div class="flex min-h-screen items-center justify-center bg-gradient-to-br from-blue-50 to-emerald-50 p-4">
    <form
      class="w-full max-w-md space-y-5 rounded-xl bg-white p-8 shadow"
      @submit.prevent="handleLogin"
    >
      <h1 class="text-center text-2xl font-bold text-gray-800">用户登录</h1>

      <!-- email -->
      <div>
        <input
          v-model="formData.email"
          type="text"
          placeholder="输入邮箱地址"
          class="w-full rounded-lg p-3 text-sm border"
          :class="emailError ? 'border-red-500' : 'border-gray-200'"
        >
        <span v-if="emailError" class="mt-1 text-xs italic text-red-500">
          {{ emailError[0] }}
        </span>
      </div>

      <!-- password -->
      <div>
        <input
          v-model="formData.password"
          type="password"
          placeholder="输入密码"
          class="w-full rounded-lg p-3 text-sm border"
          :class="passwordError ? 'border-red-500' : 'border-gray-200'"
        >
        <span v-if="passwordError" class="mt-1 text-xs italic text-red-500">
          {{ passwordError[0] }}
        </span>
      </div>

      <button
        type="submit"
        class="w-full rounded-lg bg-blue-500 px-5 py-3 text-sm font-medium text-white hover:bg-blue-600"
      >
        登录
      </button>
    </form>
  </div>
</template>
```

> ⚠️ 依赖版本差异（本节最关键差异）
>
> 课程的错误展示代码：
>
> ```ts
> // v3 写法
> const errorMap = result.error.format()
> emailError.value = errorMap.email?._errors
> ```
>
> v4 把 `error.format()` **deprecated** 了，改用顶层 `z.treeifyError(error)`，并且结构变了：
>
> ```ts
> // v4 写法
> const tree = z.treeifyError(result.error)
> // tree 结构:
> // {
> //   errors: [],
> //   properties: {
> //     email: { errors: ['Invalid email'] },
> //     password: { errors: ['Too short'] }
> //   }
> // }
> emailError.value = tree.properties?.email?.errors
> ```
>
> 区别：
>
> - v3：错误数组挂在 `_errors`（带下划线前缀）
> - v4：错误数组挂在 `errors`（无下划线），并多了 `properties` 包装层
>
> 这是 v3 → v4 升级时**最容易出 runtime 错的地方**，因为 TypeScript 不会报错（两者都是普通对象），只有运行时拿不到值才发现。

> ⚠️ Nuxt 4 差异
>
> - 课程的 `validators/user.ts` 在项目根；craft-admin 直接 `from '@my-lego/shared'`。
> - 课程的 `pages/login.vue` 在项目根；craft-admin 在 `app/pages/login.vue`。

**验证**：

1. 不填任何东西点登录 → 两个 input 红框 + 各自错误信息
2. 填非 email 字符串 + 短密码 → 同样有错
3. 填合法 email + 6 位以上密码 → 控制台输出 `valid {...}`

### 2.3 Step 3 —— 服务端：21-6 在 `server/api/users/login.post.ts` 接入校验

#### Nuxt 4 的 server 目录位置

> ⚠️ Nuxt 4 差异
> Nuxt 3 / Nuxt 4 的 `server/` 目录**都在项目根**（不下移到 `app/`），这是 Nuxt 4 故意保留的设计 —— 服务端代码独立于客户端，TypeScript 上下文也是单独的（`tsconfig.server.json`）。
>
> - 课程：`server/api/users/login.post.ts` ✓
> - craft-admin：`packages/craft-admin/server/api/users/login.post.ts` ✓（路径完全一致，只多了子包前缀）

#### 创建 API handler

```ts
// packages/craft-admin/server/api/users/login.post.ts
import { userLoginSchema } from '@my-lego/shared'

export default defineEventHandler(async (event) => {
  // h3 的 readValidatedBody：一行完成"读取 body + 校验"
  const result = await readValidatedBody(event, body => userLoginSchema.safeParse(body))

  if (!result.success) {
    throw createError({
      statusCode: 403,
      statusMessage: '验证失败',
      // v4 用 treeifyError 替代 format
      data: z.treeifyError(result.error),
    })
  }

  // 已验证数据
  const { email } = result.data

  // 21-11 / 21-12 才真正接 MongoDB + bcrypt + JWT，先返回占位数据
  return {
    username: email.split('@')[0],
    placeholder: true,
  }
})
```

需要在文件顶部 import zod：

```ts
import { z } from 'zod'
```

> 💡 h3 的 `readValidatedBody` / `getValidatedQuery` 是 Nuxt 3.7+ 引入的语法糖，签名 `(event, validator)` —— `validator` 可以是任意函数，**不限定必须用 Zod**（你可以自己写 if/else 校验也行）。但用 Zod 时配合得最自然。

### 2.4 Step 4 —— 抽离全局工具 `server/utils/runValidate.ts`

讲师的痛点：**每个 API handler 都要写"safeParse → if !success → createError → return data"四行**。注册页、修改用户、删除等都要重写一遍。抽成全局函数。

#### 创建工具

```ts
// packages/craft-admin/server/utils/runValidate.ts
import type { H3Event } from 'h3'
import type { ZodType } from 'zod'
import { z } from 'zod'

/**
 * 通用：读取 request body 并用 Zod schema 校验
 * - 失败：throw 403 createError，data 是 treeifyError 结构
 * - 成功：返回 SafeParseSuccess（含 .data 强类型）
 */
export const runValidate = async <T>(
  schema: ZodType<T>,
  event: H3Event,
) => {
  const result = await readValidatedBody(event, body => schema.safeParse(body))

  if (!result.success) {
    throw createError({
      statusCode: 403,
      statusMessage: '验证失败',
      data: z.treeifyError(result.error),
    })
  }

  return result
}
```

> 💡 关键点：泛型 `<T>` + 参数类型 `ZodType<T>` 让 `result.data` 的类型自动跟随传入 schema。
>
> - 传 `userLoginSchema`（推断出 `{ email, password }`）→ `result.data` 是 `{ email: string; password: string }`
> - 传 `userSignupSchema` → `result.data` 是 `{ email, password, confirmPwd }`
>
> 这是 Zod 文档专门有一节 ["Writing generic functions"](https://zod.dev/?id=writing-generic-functions) 讲的用法。

> ⚠️ 依赖版本差异（Zod 泛型）
>
> 课程 v3 `ZodType` 是三个泛型 `<Output, Def, Input>`；v4 简化成两个 `<Output, Input = unknown>`，并移除了 `ZodTypeAny`。
>
> - 课程的 `<T>(schema: ZodType<T>, event: H3Event)` 在 v4 也能跑（默认 Input 是 unknown，对运行时无影响）。
> - 如果你看到老代码用 `ZodTypeAny`，v4 改成 `ZodType` 即可。

> ⚠️ Nuxt 4 差异
>
> - 课程 `server/utils/runValidate.ts` 在项目根；craft-admin 在 `packages/craft-admin/server/utils/runValidate.ts`。
> - Nuxt 自动注册 `server/utils/` 下导出的所有函数为全局可用（不需要 import）—— 这条对 Nuxt 3 / Nuxt 4 都成立。

#### 重写 login.post.ts

```ts
// packages/craft-admin/server/api/users/login.post.ts
import { userLoginSchema } from '@my-lego/shared'

export default defineEventHandler(async (event) => {
  // 一行搞定：读取 body + 校验 + 错误处理
  const result = await runValidate(userLoginSchema, event)

  const { email } = result.data

  // 21-11 接 MongoDB / bcrypt / JWT
  return {
    username: email.split('@')[0],
    placeholder: true,
  }
})
```

代码从 14 行精简到 9 行，登录、注册、改密码所有 handler 都能复用 `runValidate`。

## 4. 易错点 + 关键 API 速查

### 4.1 易错点 Checklist

1. **`z.treeifyError` 结构比 `error.format()` 多一层**。v3 的 `error.format().email._errors` ≈ v4 的 `z.treeifyError(err).properties.email.errors`。跨版本迁移时最容易踩。
2. **`error.format()` 在 v4 仍然能跑（兼容性保留）**。如果只是临时跟敲课程代码，写 `error.format()` 也不会报错；但 v4 已标记 deprecated，建议直接用 `treeifyError`。
3. **Nuxt 4 `server/` 不在 `app/` 下**。不要被 21-3 / 21-5 的 `app/pages/` `app/layouts/` 误导，去 `app/server/api/...` 找文件。
4. **`server/utils/runValidate.ts` 是全局自动导入**。在 API handler 里写 `runValidate(...)` 不需要 import；但**类型定义** `H3Event` / `ZodType` 还是要 import（运行时和编译时区别）。
5. **`readValidatedBody` 是 h3 提供的全局函数**，Nuxt 自动注入。同类还有 `getValidatedQuery` / `getValidatedRouterParams`，按需用。
6. **客户端 import shared schema 不会泄漏 server 代码**。`@my-lego/shared` 是纯 TS lib，没有 Node-only 依赖；如果未来 schema 文件里 `import { readFileSync } from 'fs'` 就会让客户端打包失败 —— 保持 schema 纯净。
7. **alias 子路径 import 失败**。craft-admin 的 alias 只配了入口 `@my-lego/shared` → `index.ts`。写 `from '@my-lego/shared/validators/user'` 会报"找不到模块"，必须从 `'@my-lego/shared'` 顶层 import。
8. **服务端 schema 校验的 `400 vs 403`**。课程用 `403`（语义其实是"无权限"），更标准是 `422 Unprocessable Entity`（数据格式不对）或 `400 Bad Request`。本节遵循课程用 403，后续接 NestJS 时再统一规范。

### 4.2 关键 API 速查

| 项目                                                  | 作用                                | 类型                                         |
| ----------------------------------------------------- | ----------------------------------- | -------------------------------------------- |
| `userLoginSchema.safeParse(data)`                     | 客户端校验                          | 客户端 / 同构                                 |
| `userLoginSchema.parse(data)`                         | 客户端校验（抛错）                  | 客户端 / 同构                                 |
| `z.treeifyError(error)` (v4)                          | 错误转嵌套对象，按字段取            | 客户端 / 服务端 / 同构                        |
| `z.prettifyError(error)` (v4)                         | 错误转可读多行字符串                | 调试用                                       |
| `error.format()` (v3, deprecated in v4)               | 同 `treeifyError` 但结构不同         | 仅向后兼容                                   |
| `readValidatedBody(event, validator)`                  | h3 提供，读 body + 校验            | 服务端                                       |
| `getValidatedQuery(event, validator)`                  | h3 提供，读 query + 校验           | 服务端                                       |
| `getValidatedRouterParams(event, validator)`           | h3 提供，读路由参数 + 校验         | 服务端                                       |
| `defineEventHandler(handler)`                          | 定义 API handler                   | 服务端                                       |
| `createError({ statusCode, statusMessage, data })`     | 抛带状态码的错误                   | 服务端                                       |
| `runValidate(schema, event)`                           | 自封装：读 body + Zod 校验 + 抛 403 | 服务端（craft-admin/server/utils 自动导入）   |

### 4.3 与课程相比，本节**改写**了什么

- Schema 位置 `validators/user.ts` → **`@my-lego/shared`**（monorepo 红利）
- 客户端错误格式化 `error.format()._errors` → **`z.treeifyError(error).properties.errors`**（zod v4）
- 错误参数 `{ message: '...' }` → **`{ error: '...' }`**（zod v4）
- 服务端 `runValidate` 实现：去掉 `format()`，改 `treeifyError`；保持泛型签名

## 1. 概述

21-4 学完了 Zod 的 API，本节把 Zod 真正接入登录流程：

- **21-5（客户端）**：在 `login.vue` 用 `reactive` form data 配合 `safeParse` 做提交前校验；用 `treeifyError` 把错误格式化成对象，按字段显示在每个 input 下方。

> 🧭 **两个 "shared" 千万别混淆**
>
> | | Nuxt 4 项目级 `shared/`（本节用这个） | monorepo 的 `@my-lego/shared` 包 |
> | --- | --- | --- |
> | 位置 | `packages/craft-admin/shared/`（与 `app/`、`server/` 平级） | `packages/shared/`（独立 workspace 包） |
> | 引用方式 | `#shared/...`（Nuxt 自动配置的别名） | `import ... from '@my-lego/shared'`（nuxt.config 里配的 alias） |
> | 共享范围 | craft-admin 内部的 app ↔ server | 整个 monorepo 的所有子包（craft / backend / admin） |
> | 适用场景 | 只有 admin 自己用的逻辑 | 真正跨项目复用的工具 |
>
> 登录 schema 选 **Nuxt 4 项目级 `shared/`**：只有 admin 用，没必要上升到跨包共享，避免过度抽象。

## 2. 知识点 + 演示复盘

### 2.1 先认识 Nuxt 4 的 `shared/` 目录

`shared/` 是 Nuxt 3.14+ / Nuxt 4 引入的项目级目录，专门放**客户端和服务端都要用**的纯逻辑代码。

规则（务必记住）：

- 位置在**项目根**（craft-admin 下），与 `app/`、`server/` 平级 —— **不在 `app/` 里**。
- **只有 `shared/utils/` 和 `shared/types/` 会被自动导入**。
- 其他子目录（如我们的 `shared/validators/`）里的文件，要用 **`#shared/...`** 别名手动 import。
- `shared/` 里的代码 **不能 import 任何 Vue 或 Nitro 专属 API**（它要同时跑在两端）。`zod` 是纯 JS 库，放心用。

为什么 schema 放 `shared/validators/` 而不是 `shared/utils/`？—— 语义更清晰（validators 是一类），并且我们本来就要手动 import（`#shared/validators/user`），不依赖自动导入。

> ⚠️ Nuxt 4 差异
> 课程（Nuxt 3）把 schema 放项目根 `validators/user.ts`，客户端和服务端各自相对路径 import。
> craft-admin 放 `shared/validators/user.ts`，两端统一用 `#shared/validators/user`，是 Nuxt 4 推荐的"app/server 共享"方式。

### 2.2 Step 1 —— 定义 schema

#### 在 craft-admin 安装 zod（21-4 已装则跳过）

```bash
pnpm --filter @my-lego/craft-admin add zod
```

> ⚠️ Monorepo 差异
> zod 装在 **craft-admin 自己的依赖**里（因为 `shared/` 是 craft-admin 的项目级目录，不是独立包）。不要装到 monorepo 根，也不需要装到 `@my-lego/shared` 包。

#### 新建 `packages/craft-admin/shared/validators/user.ts`

```ts
// packages/craft-admin/shared/validators/user.ts
import { z } from 'zod'

export const userLoginSchema = z.object({
  email: z.email(), // v4 顶层写法（v3 是 z.string().email()）
  password: z.string().min(6),
})

export type UserLoginType = z.infer<typeof userLoginSchema>

// 注册 schema 一并定义好（21-10 注册页会用到，含 refine 两次密码一致）
export const userSignupSchema = z.object({
  email: z.email(),
  password: z.string().min(6),
  confirmPwd: z.string().min(6),
}).refine(data => data.password === data.confirmPwd, {
  error: 'Password must match!', // v4 用 error 而不是 message
  path: ['confirmPwd'],
})

export type UserSignupType = z.infer<typeof userSignupSchema>
```

> ⚠️ 依赖版本差异
>
> - 课程 v3：`z.string().email()`、`refine` 的第二参用 `{ message: '...' }`。
> - craft-admin v4：`z.email()`、`refine` 用 `{ error: '...' }`（`message` 已弃用）。
> - `refine` 用法 21-10 详讲，这里先把 signup schema 写好。

### 2.3 Step 2 —— 客户端校验（21-5）

> 💡 **关于这一版代码**：21-5 视频用 `reactive` + `safeParse` 手动管理错误，是为了让你理解"不借助任何表单库时 Zod 怎么用"。craft-admin 最终的 `login.vue` 是 21-9 的 **VeeValidate + ValidateInput** 版（更简洁），所以**这一版 reactive 写法你跑通理解即可，不必作为最终提交**。下面仍按视频还原。
>
> 这里也**不复制 HyperUI 模板**（见 21-2 的决定），用最简洁的手写表单。

```vue
<!-- packages/craft-admin/app/pages/login.vue（21-5 阶段，理解用，最终见 21-7_8_9）-->
<template>
  <div class="flex min-h-screen items-center justify-center bg-linear-to-br from-blue-50 to-emerald-50 p-4">
    <form
      class="w-full max-w-md space-y-5 rounded-xl bg-white p-8 shadow"
      @submit.prevent="handleLogin"
    >
      <h1 class="text-center text-2xl font-bold text-gray-800">用户登录</h1>

      <div>
        <input
          v-model="formData.email"
          type="text"
          placeholder="输入邮箱地址"
          class="w-full rounded-lg p-3 text-sm border"
          :class="emailError ? 'border-red-500' : 'border-gray-200'"
        >
        <span v-if="emailError" class="mt-1 text-xs italic text-red-500">{{ emailError[0] }}</span>
      </div>

      <div>
        <input
          v-model="formData.password"
          type="password"
          placeholder="输入密码"
          class="w-full rounded-lg p-3 text-sm border"
          :class="passwordError ? 'border-red-500' : 'border-gray-200'"
        >
        <span v-if="passwordError" class="mt-1 text-xs italic text-red-500">{{ passwordError[0] }}</span>
      </div>

      <button
        type="submit"
        class="w-full rounded-lg bg-blue-500 px-5 py-3 text-sm font-medium text-white hover:bg-blue-600"
      >
        登录
      </button>
    </form>
  </div>
</template>

<script setup lang="ts">
import { userLoginSchema, type UserLoginType } from '#shared/validators/user'
import { z } from 'zod'

definePageMeta({ layout: 'empty' })

const formData = reactive<UserLoginType>({
  email: '',
  password: '',
})

const emailError = ref<string[] | undefined>(undefined)
const passwordError = ref<string[] | undefined>(undefined)

const handleLogin = () => {
  const result = userLoginSchema.safeParse(formData)

  if (!result.success) {
    // ⚠️ v4 用 treeifyError 替代 v3 的 error.format()
    const tree = z.treeifyError(result.error)
    emailError.value = tree.properties?.email?.errors
    passwordError.value = tree.properties?.password?.errors
    return
  }

  emailError.value = undefined
  passwordError.value = undefined
  console.log('valid', result.data)
}
</script>
```

> ⚠️ 依赖版本差异（本节最关键的 runtime 坑）
>
> 课程 v3 的错误格式化：
>
> ```ts
> const errorMap = result.error.format()
> emailError.value = errorMap.email?._errors   // 注意 _errors（带下划线）
> ```
>
> v4 把 `error.format()` **deprecated**，改用顶层 `z.treeifyError(error)`，**结构也变了**：
>
> ```ts
> const tree = z.treeifyError(result.error)
> // {
> //   errors: [],
> //   properties: {
> //     email:    { errors: ['Invalid email'] },
> //     password: { errors: ['Too small: ...'] }
> //   }
> // }
> emailError.value = tree.properties?.email?.errors  // errors（无下划线）+ 多一层 properties
> ```
>
> - v3：`errorMap.email._errors`
> - v4：`tree.properties.email.errors`
>
> TypeScript 不会报错（都是普通对象），只有运行时取不到值才发现 —— 升级时最容易踩。

> ⚠️ Nuxt 4 差异
>
> - schema：`import { userLoginSchema } from '#shared/validators/user'`（不是相对路径，也不是 `@my-lego/shared`）。
> - 页面：`app/pages/login.vue`；布局 `definePageMeta({ layout: 'empty' })`。
> - eslint `block-order`：`<template>` 在 `<script>` 之前。

**验证**：空表单点登录 → 两个红框 + 错误；非法 email + 短密码 → 有错；合法值 → 控制台 `valid {...}`。

### 2.4 Step 3 —— 服务端校验（21-6 第一版）

#### Nuxt 4 的 server 目录位置

> ⚠️ Nuxt 4 差异
> Nuxt 3 / Nuxt 4 的 `server/` 目录**都在项目根**（不下移到 `app/`），服务端代码独立、TypeScript 上下文也独立。
>
> - 课程：`server/api/users/login.post.ts`
> - craft-admin：`packages/craft-admin/server/api/users/login.post.ts`（路径一致，只多子包前缀）

```ts
// packages/craft-admin/server/api/users/login.post.ts（第一版，未抽 runValidate）
import { userLoginSchema } from '#shared/validators/user'
import { z } from 'zod'

export default defineEventHandler(async (event) => {
  // h3 的 readValidatedBody：一行完成"读取 body + 校验"
  const result = await readValidatedBody(event, body => userLoginSchema.safeParse(body))

  if (!result.success) {
    throw createError({
      statusCode: 403,
      message: '验证失败',
      data: z.treeifyError(result.error),
    })
  }

  const { email } = result.data

  // 21-11 / 21-12 才真正接 MongoDB + bcrypt + JWT，先返回占位数据
  return {
    username: email.split('@')[0],
    placeholder: true,
  }
})
```

> 💡 `readValidatedBody(event, validator)` 是 Nuxt/h3 提供的全局函数（自动注入），`validator` 是任意函数，不限定 Zod。同类还有 `getValidatedQuery`、`getValidatedRouterParams`。

> ⚠️ 依赖版本差异 + 一个生产环境坑（`message` vs `statusMessage`）
>
> - 课程用 `statusMessage: '验证失败'` + `data: result.error.format()`。
> - craft-admin 用 `message: '验证失败'` + `data: z.treeifyError(...)`。
> - **注意**：Nitro 生产模式默认**屏蔽 `message`**（防止泄漏内部错误），但保留 `statusMessage` 和 `data`。本节验证错误是给开发看的、字段错误走 `data`（treeify），所以用 `message` 没问题；但**面向用户展示的提示**（如 21-11 的"邮箱已注册"）建议用 `statusMessage` 或塞进 `data`，否则上线后取不到。

#### 客户端临时联调

把 `login.vue` 的 `handleLogin` 改成异步发请求（21-8 接 VeeValidate 后会正式整合）：

```ts
const handleLogin = async () => {
```

### 2.5 Step 4 —— 抽离全局工具 `server/utils/runValidate.ts`

痛点：每个 API handler 都要写"safeParse → if !success → createError → return"四步。注册、改密码等都要重写。抽成全局函数。

```ts
// packages/craft-admin/server/utils/runValidate.ts
import type { H3Event } from 'h3'
import type { ZodType } from 'zod'
import { z } from 'zod'

/**
 * 通用：读取 request body 并用 Zod schema 校验
 * - 失败：throw 403 createError，data 是 treeifyError 结构
 * - 成功：返回 SafeParseSuccess（含 .data 强类型）
 */
export const runValidate = async <T>(
  schema: ZodType<T>,
  event: H3Event,
) => {
  const result = await readValidatedBody(event, body => schema.safeParse(body))

  if (!result.success) {
    throw createError({
      statusCode: 403,
      message: '验证失败',
      data: z.treeifyError(result.error),
    })
  }

  return result
}
```

> 💡 关键点：泛型 `<T>` + 参数 `ZodType<T>` 让 `result.data` 类型自动跟随传入 schema：
>
> - 传 `userLoginSchema` → `result.data` 是 `{ email: string; password: string }`
> - 传 `userSignupSchema` → `result.data` 是 `{ email; password; confirmPwd }`
>
> Zod 文档 ["Writing generic functions"](https://zod.dev/api#writing-generic-functions) 专门讲这个。

> ⚠️ 依赖版本差异（Zod 泛型）
> 课程 v3 `ZodType` 是 `<Output, Def, Input>` 三个泛型；v4 简化为 `<Output, Input = unknown>`，并移除 `ZodTypeAny`。课程的 `<T>(schema: ZodType<T>, event)` 在 v4 照常工作。

> ⚠️ Nuxt 4 差异（自动导入 vs 显式 import）
> `server/utils/` 下的导出**默认会被 Nitro 自动导入**，理论上在 handler 里直接写 `runValidate(...)` 即可。
> 但 craft-admin 实际选择**显式 import**，IDE 跳转/类型更稳：
>
> ```ts
> import { runValidate } from '#server/utils/runValidate'
> ```
>
> `#server` 是 Nuxt 4 给 `server/` 目录配置的别名。两种方式都行，显式更清晰。

#### 重写 login.post.ts

```ts
// packages/craft-admin/server/api/users/login.post.ts（最终版）
import { runValidate } from '#server/utils/runValidate'
import { userLoginSchema } from '#shared/validators/user'

export default defineEventHandler(async (event) => {
  // 一行搞定：读取 body + 校验 + 错误处理
  const result = await runValidate(userLoginSchema, event)

  const { email } = result.data

  // 21-11 接 MongoDB / bcrypt / JWT
  return {
    username: email.split('@')[0],
    placeholder: true,
  }
})
```

## 4. 易错点 + 关键 API 速查

### 4.1 易错点 Checklist

1. **`z.treeifyError` 结构比 `error.format()` 多一层 `properties`、且字段从 `_errors` 变 `errors`**。v3→v4 迁移最容易踩，TS 不报错只能 runtime 发现。
2. **`error.format()` 在 v4 仍能跑（兼容保留）**，但已 deprecated，建议直接用 `treeifyError`。
3. **Nuxt 4 `server/` 不在 `app/` 下**。别被 `app/pages/`、`app/layouts/` 误导。
4. **`shared/` 里不能 import Vue / Nitro 代码**。否则两端共享会编译失败。schema 只 import zod（纯 JS），安全。
5. **`shared/validators/` 不自动导入**，必须 `#shared/validators/user` 手动 import（只有 `shared/utils`、`shared/types` 自动导入）。
6. **`createError` 的 `message` 生产环境会被屏蔽**。面向用户的提示用 `statusMessage` 或放 `data`；`message` 仅适合开发期调试信息。
7. **`readValidatedBody` 是全局函数**，handler 里直接用、无需 import；但 `H3Event` / `ZodType` 这些**类型**要 import。
8. **客户端也能 import `#shared`**。`#shared/validators/user` 在 `app/` 和 `server/` 都能用，这正是 shared 目录的意义。

### 4.2 关键 API 速查

| 项目                                                  | 作用                                | 端                                          |
| ----------------------------------------------------- | ----------------------------------- | -------------------------------------------- |
| `userLoginSchema.safeParse(data)`                     | 校验，返回 `{ success, data?, error? }` | 客户端 / 服务端 / 同构                       |
| `z.treeifyError(error)` (v4)                          | 错误转嵌套对象，按字段取            | 同构                                         |
| `z.prettifyError(error)` (v4)                         | 错误转可读多行字符串                | 调试                                         |
| `readValidatedBody(event, validator)`                  | h3：读 body + 校验                  | 服务端                                       |
| `getValidatedQuery(event, validator)`                  | h3：读 query + 校验                 | 服务端                                       |
| `defineEventHandler(handler)`                          | 定义 API handler                   | 服务端                                       |
| `createError({ statusCode, statusMessage, message, data })` | 抛带状态码的错误             | 服务端                                       |
| `runValidate(schema, event)`                           | 自封装：读 body + 校验 + 抛 403      | 服务端（`#server/utils/runValidate`）         |
| `#shared/...`                                          | 引用 Nuxt 4 项目级 `shared/` 文件   | 客户端 / 服务端                              |
| `#server/...`                                          | 引用 `server/` 目录文件             | 服务端                                       |

### 4.3 与课程相比，本节**改写**了什么

## 1. 概述

### 1.1 为什么需要"数据验证库"

接下来要做登录功能，**永远不要相信用户的输入**：

- **前端**：约束用户输入符合规则，提交前先给提示。
- **后端**：再次校验 API 入参，防止前端绕过。

讲师指出之前方案的两个痛点：

1. **前后端用了完全不同的方案**（前端 antd Form 组件、后端 ajv）—— 一套规则被写两遍，且和具体框架强耦合，迁移、自定义、跨端共享都麻烦。
2. **TypeScript 与运行时验证脱节** —— TS 是静态类型，运行时拿不到；同一份数据结构往往要写一遍 `interface` 再写一遍 schema。

### 1.2 Zod 的解法

[Zod](https://zod.dev/) 一次性解决两个问题：

1. **一套 schema，前后端共用**。Nuxt 是全栈框架，schema 写一份就行。
2. **schema 直接推断 TS 类型**（`z.infer<typeof X>`），再也不重复写 interface。

特性速览：

- 声明式 API、链式调用、组合验证规则
- 内置数十种字符串/数字/日期/对象/数组验证器
- 与 vee-validate、react-hook-form、tRPC、drizzle 等都有集成

### 1.3 本节产出

本节是 **Zod 纯演示**，目标是把 Zod 的核心 API 跑一遍：

1. 安装 Zod
2. `z.string()` / `.email()` / `.min()` 链式约束
3. `parse` 与 `safeParse` 两种校验入口
4. 错误结构（`ZodError.issues`、`treeifyError`）
5. 自定义错误信息
6. `z.object` 复合 schema
7. `z.infer` 推断 TS 类型

下一节 21-5 才把 Zod 接入登录页。

### 2.2 最基本的 schema：`z.string()`

讲师在 `pages/index.vue` 演示。我们把演示放到 craft-admin 上一节刚建的 `app/pages/index.vue`：

```vue
<!-- packages/craft-admin/app/pages/index.vue -->
<script setup lang="ts">
import { z } from 'zod'

// 定义一个 schema：希望这个值是 string
const mySchema = z.string()

// parse：成功返回值，失败抛错
const result = mySchema.parse('wikim')
console.log(result) // → 'wikim'

// 失败演示：传入数字会抛 ZodError
// const result2 = mySchema.parse(12)
</script>
```

打开浏览器（开发者控制台 + Nuxt SSR 终端都能看到），传 string 时输出原值；传 12（number）会得到 `ZodError`：「Expected string, received number」。

### 2.3 不抛错的校验：`safeParse`

```ts
const mySchema = z.string()
const result = mySchema.safeParse(12)
console.log(result)
// → { success: false, error: ZodError }

if (!result.success) {
  console.log(result.error.issues)
}
```

返回对象，`success` 为 boolean，失败时 `error.issues` 是错误数组。这是讲师**强烈推荐**的方式（"我经常会使用的"），因为不需要 `try / catch`、TypeScript 通过 **type guard** 自动收窄类型：

- `if (result.success)` 内：`result.data` 是已验证的强类型值
- `else` 内：`result.error` 才有定义

### 2.4 链式调用

```ts
// 必须是 email
z.string().email()                  // v3 写法（v4 仍兼容但已 deprecated）
z.email()                           // v4 顶层写法

// 必须是 email，且自定义错误信息
z.string().email({ message: '输入Email' })       // v3 / v4 兼容写法
z.email({ error: '输入Email' })                  // v4 推荐写法

// 多个约束串起来
z.string().email().min(5)
```

> ⚠️ 依赖版本差异（v3 → v4 API 重命名）
>
> v4 把所有"字符串格式校验"提到顶层并 deprecated 旧链式 API：
>
> | v3 写法                  | v4 推荐写法    | 备注                           |
> | ------------------------ | -------------- | ------------------------------ |
> | `z.string().email()`     | `z.email()`    | 链式仍可用，但已 deprecated     |
> | `z.string().uuid()`      | `z.uuid()`     | 同上                           |
> | `z.string().url()`       | `z.url()`      | 同上                           |
> | `z.string().datetime()`  | `z.iso.datetime()` | 同上                       |
> | `z.string().ip()`        | `z.ipv4()` / `z.ipv6()` | 拆成两个                |
> | `{ message: '...' }`     | `{ error: '...' }` | 错误参数统一改名             |
>
> **lego-admin 课程项目的写法是 v3 风格**（见 `validators/user.ts`）。craft-admin 推荐用 v4 顶层 API，原因：① 更短；② tree-shaking 友好（v4 把每个 format 分成独立 class，bundle 更小）；③ v5 会移除 deprecated 写法，早点适配早安心。
>
> **偷懒方案**：装好 zod@4 后跑 `npx @zod/codemod --transform v3-to-v4 ./src` 自动改写 v3 代码。21-5 我们手写 schema 时直接按 v4 风格写即可。

### 2.5 错误对象的结构

```ts
const mySchema = z.string().email({ message: 'invalid email' }).min(5)
const result = mySchema.safeParse('a')

if (!result.success) {
  console.log(result.error.issues)
  /*
    [
      { code: 'invalid_string', validation: 'email', message: 'invalid email', path: [] },
      { code: 'too_small',      minimum: 5, message: 'String must contain at least 5 character(s)', path: [] }
    ]
  */
}
```

每个 issue 包含：

- `code`：错误码（`invalid_string` / `too_small` / `invalid_type` …）
- `path`：出错字段路径（对象/嵌套时有用）
- `message`：可直接显示给用户的文案

### 2.6 `z.object()` 复合 schema（嵌套）

```ts
// 等价于课程里的 userLoginSchema
const loginSchema = z.object({
  email: z.email(),                    // v4 顶层写法
  password: z.string().min(6),
})
```

`z.object` 内层支持任意嵌套（再嵌 `z.object` / `z.array` / `z.union` …）。

### 2.7 自动推断 TS 类型 `z.infer`

```ts
type LoginData = z.infer<typeof loginSchema>
// → { email: string; password: string }
```

WebStorm 把鼠标移到 `LoginData` 上能看到展开后的类型 —— **不需要手写一遍 interface**，schema 改了类型同步变。这就是讲师强调的 Zod 第二大杀手锏。

> 💡 进阶：`z.input<typeof X>` 与 `z.output<typeof X>` —— 当 schema 含 `.transform()` 时输入输出类型不同，可分别取。21-4 演示用不到，存档备查。

### 2.8 完整演示代码（在 craft-admin 跑一遍）

```vue
<!-- packages/craft-admin/app/pages/index.vue -->
<script setup lang="ts">
import { z } from 'zod'

// 1) 最基本
const stringSchema = z.string()
console.log('parse:', stringSchema.parse('wikim'))

const r1 = stringSchema.safeParse(12)
if (!r1.success) {
  console.log('error.issues:', r1.error.issues)
}

// 2) 链式 + email + min
const emailSchema = z.email({ error: '输入Email' }).min(5)
const r2 = emailSchema.safeParse('a')
if (!r2.success) console.log('email issues:', r2.error.issues)

// 3) z.object + z.infer
const loginSchema = z.object({
  email: z.email(),
  password: z.string().min(6),
})

type LoginData = z.infer<typeof loginSchema>

const r3 = loginSchema.safeParse({ email: 'a@b.com', password: '123456' })
console.log('object safeParse:', r3)
</script>

<template>
  <div class="rounded-lg bg-white p-8 shadow-xs">
    <h2 class="text-2xl font-bold text-gray-800">Welcome to craft-admin</h2>
    <p class="mt-2 text-gray-500">打开控制台看 Zod 演示输出。</p>
  </div>
</template>
```

刷新 `/` 查看 console 与 SSR 终端，所有验证场景都能看到对应结果。

## 4. 易错点 + 关键 API 速查

### 4.1 易错点 Checklist

1. **`parse` 与 `safeParse` 二选一**。`parse` 抛错（适合"必须成功"的场景）；`safeParse` 返回 `{ success, data, error }`（适合"友好展示错误"的场景）。99% 业务用 `safeParse`。
2. **必须先 `if (!result.success)` 再访问 `error`**。TypeScript 通过 type guard 收窄类型 —— 跳过判断直接 `result.error.issues` 会报类型错。
3. **不要混用 v3 / v4 写法**。一个项目里一致用一种风格，特别是 `error` vs `message` 字段名。
4. **`z.email()` 的默认 regex 是 Gmail 风格的严格规则**。如果你想要更宽松的（比如允许 unicode、+号等），用 `z.email({ pattern: z.regexes.unicodeEmail })`。
5. **`z.object({...}).strict()` vs `.passthrough()`**。默认（v4：`.loose()`）会"剥离"多余字段；`.strict()` 多余字段直接报错；`.passthrough()` 保留多余字段。课程没演示，写 schema 时根据"是否允许扩展字段"选。
6. **TypeScript 类型推断失败**：检查 schema 是否用 `as const`、是否 export，以及 `tsconfig.json` 的 `strict` 是否开启。

### 4.2 关键 API 速查

| API                           | 作用                                          |
| ----------------------------- | --------------------------------------------- |
| `z.string()` / `z.number()` / `z.boolean()` / `z.date()` | 原始类型             |
| `z.email()` / `z.url()` / `z.uuid()` (v4)           | 字符串格式校验（v3 是链式 `.email()` 等） |
| `.min(n)` / `.max(n)` / `.length(n)` / `.regex(pattern)` | 长度/格式约束                      |
| `.optional()` / `.nullable()` / `.default(v)` | 可选/可空/默认值                         |
| `z.object({...})` / `z.array(s)` / `z.union([s1, s2])` / `z.tuple([s1, s2])` | 组合 schema |
| `schema.parse(v)`             | 校验失败抛 `ZodError`                          |
| `schema.safeParse(v)`         | 校验返回 `{ success, data?, error? }`          |
| `schema.parseAsync(v)` / `.safeParseAsync(v)` | 异步 schema（含 `.refine` 异步逻辑）  |
| `z.infer<typeof schema>`      | 推断 TypeScript 类型                           |
| `z.input<typeof schema>` / `z.output<typeof schema>` | `.transform()` 输入输出类型分别提取  |
| `error.issues`                | `ZodError` 的错误数组（v3/v4 均可用）           |
| `error.format()` (v3) / `z.treeifyError(error)` (v4) | 转换成嵌套对象，便于按字段显示       |
| `z.prettifyError(error)` (v4) | 转换成多行可读字符串                            |

### 4.3 与课程相比，本节**省略或改写**了什么

- 课程在 `validators/user.ts` 里写 `z.string().email()` (v3 风格) —— craft-admin 用 `z.email()` (v4 顶层)。
- 课程没有显式提到 `.format()` 是 deprecated —— v4 视角下要标记，21-5 / 21-6 实际用到时给 `z.treeifyError(error)` 的对照。
- 课程没演示 `parseAsync` / `prettifyError` —— 留作速查。

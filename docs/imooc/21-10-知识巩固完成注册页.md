## 2. 知识点 + 演示复盘

### 2.1 新知识点：Zod `.refine()` 自定义校验

`z.object({...})` 里每个字段的规则是**独立**的。但"password 和 confirmPwd 相等"需要**同时看两个字段** —— 这种跨字段逻辑用 `.refine()`。

```ts
// packages/craft-admin/shared/validators/user.ts（21-5 已定义，这里详解）
import { z } from 'zod'

export const userSignupSchema = z.object({
  email: z.email(),
  password: z.string().min(6),
  confirmPwd: z.string().min(6),
}).refine(
  // 第 1 个参数：校验函数，返回 true 通过、false 不通过
  data => data.password === data.confirmPwd,
  // 第 2 个参数：配置
  {
    error: 'Password must match!', // v4 用 error（v3 是 message）
    path: ['confirmPwd'], // 错误归属到哪个字段
  },
)

export type UserSignupType = z.infer<typeof userSignupSchema>
```

要点：

- `.refine(checkFn, options)` 链在 `z.object({...})` **后面**（作用于整个对象，能拿到所有字段）。
- `checkFn(data)` 的 `data` 就是整个对象（类型由前面的 object 推断）。
- `path: ['confirmPwd']` —— 决定错误**挂到哪个字段**。这样 `<ValidateInput name="confirmPwd">` 才能显示这条错误（不写 path 的话错误挂在对象根，字段级 input 拿不到）。

> ⚠️ 依赖版本差异（refine v3 → v4）
>
> | | v3 | v4 |
> | --- | --- | --- |
> | 错误信息字段 | `{ message: '...' }` | `{ error: '...' }` |
> | 多步自定义校验 | `.superRefine((val, ctx) => ...)` | 改名 `.check((ctx) => ...)`（`superRefine` 已 deprecated） |
> | `.refine()` 本身 | 支持 | 支持（仅 `message` → `error`） |
>
> 课程是 `{ message: "Passwords don't match" }`；craft-admin 是 `{ error: 'Password must match!' }`。

> 💡 进阶（本节用不到）：`.refine` 只能返回一条错误、挂一个 path。要在一次校验里挂多条错误 / 多个 path，用 v4 的 `.check()`（v3 的 `superRefine`）。

### 2.2 写 `signup.vue`（复用 ValidateInput）

跟 `login.vue` 几乎一样，多一个 `confirmPwd` 字段，schema 换成 `userSignupSchema`。

```vue
<!-- packages/craft-admin/app/pages/signup.vue -->
<template>
  <div class="flex min-h-screen items-center justify-center bg-linear-to-br from-blue-50 to-emerald-50 p-4">
    <form
      class="w-full max-w-md space-y-5 rounded-xl bg-white p-8 shadow"
      @submit="handleSignup"
    >
      <h1 class="text-center text-2xl font-bold text-gray-800">注册成为会员</h1>
      <p class="text-center text-sm text-gray-500">输入以下信息完成注册</p>

      <ValidateInput name="email" placeholder="输入电子邮箱地址" />
      <ValidateInput name="password" type="password" placeholder="输入密码" />
      <ValidateInput name="confirmPwd" type="password" placeholder="再次输入密码" />

      <div class="flex items-center justify-between">
        <p class="text-sm text-gray-500">
          已经有账户了？
          <NuxtLink to="/login" class="underline">登录</NuxtLink>
        </p>
        <button
          type="submit"
          :disabled="isSubmitting"
          class="rounded-lg bg-blue-500 px-5 py-3 text-sm font-medium text-white hover:bg-blue-600 disabled:opacity-75"
        >
          {{ isSubmitting ? '读取中...' : '注册' }}
        </button>
      </div>
    </form>
  </div>
</template>

<script setup lang="ts">
import { userSignupSchema } from '#shared/validators/user'
import { toTypedSchema } from '@vee-validate/zod'
import { useForm } from 'vee-validate'

definePageMeta({ layout: 'empty' })

const { handleSubmit, isSubmitting } = useForm({
  validationSchema: toTypedSchema(userSignupSchema),
})

const handleSignup = handleSubmit(async (values) => {
  // values 强类型 { email; password; confirmPwd }
  console.log('signup values', values)
  // 21-11 接后端 /api/users/signup
})
</script>
```

要点：

- `<ValidateInput name="confirmPwd" type="password" />` 的 `name` **必须**等于 schema 里的 `confirmPwd`，refine 的错误（`path: ['confirmPwd']`）才能显示在这个 input 下。
- 三个字段全部复用 `ValidateInput` —— 这正是 21-9 抽组件的回报：注册页几乎零样板。

### 2.4 验证

1. 访问 `/signup`，走 `empty` 布局（无 Header）。
2. 三字段都空 → 点注册 → 三条 required 错误。
3. email 合法、password 填 `123456`、confirmPwd 填 `123457` → **confirmPwd 下显示 `Password must match!`**（refine 生效）。

5. `/login` ↔ `/signup` 通过 NuxtLink 互跳。

## 4. 易错点 + 关键 API 速查

### 4.1 易错点 Checklist

1. **`.refine()` 要链在 `z.object({...})` 之后**，不能链在单个字段上（单字段拿不到其它字段值）。
2. **`path: ['confirmPwd']` 不写的话**，错误挂在对象根（`treeifyError().errors`），`<ValidateInput name="confirmPwd">` 显示不出来。
3. **v4 用 `error` 不是 `message`**。写 `message` 不报错但被忽略（v4 已弃用该字段名）。
4. **`name="confirmPwd"` 拼写要和 schema key 完全一致**（驼峰），否则错误绑不上、提交值缺字段。
5. **refine 的触发时机**：它是整体 object 校验，所以单独改 password 时，confirmPwd 的"一致性"错误可能要等 confirmPwd 也被 touch / 表单提交才刷新 —— 这是正常现象。
6. **后端用不用 confirmPwd**：confirmPwd 只是前端体验，后端注册其实只需要 email + password。21-11 会讨论后端要不要复用含 confirmPwd 的 schema。

### 4.2 关键 API 速查

| API                                         | 作用                                  |
| ------------------------------------------- | ------------------------------------- |
| `z.object({...}).refine(fn, { error, path })` | 跨字段自定义校验                      |
| `z.object({...}).check(ctx => ...)` (v4)    | 多错误 / 多 path 的高级自定义校验      |
| `path: ['confirmPwd']`                      | 把 refine 错误归属到指定字段          |
| `toTypedSchema(userSignupSchema)`           | Zod → vee-validate + 类型推断          |
| `<ValidateInput name="confirmPwd" />`       | 复用组件，name 对应 schema key         |

### 4.3 与课程相比，本节**改写**了什么

- refine 错误参数 `{ message }` → `{ error }`（zod v4）
- schema 来源 `~/validators/user` → `#shared/validators/user`
- 布局 `custom` → `empty`
- 复用 21-9 的 `ValidateInput`（与课程一致，但路径在 `app/components/`）
- 不复制 HyperUI，手写表单；Vue 块 template 在前
